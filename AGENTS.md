# 智能体工作入口

本仓库是一个工具箱项目，暂定包含后台服务端、Web 端和 App 端。当前项目框架未定，代码与目录以占位和约束为主。

## 先读地图

开始任何实现前，先阅读这些文件：

- `docs/CONSTRAINTS.md`：项目硬约束与工程纪律。
- `docs/ARCHITECTURE.md`：暂定架构边界与依赖方向。
- `docs/QUALITY.md`：质量门禁、测试与验证要求。
- `docs/GIT.md`：Git 提交、推送与智能体提交纪律。
- `docs/adr/0001-harness-engineering-constraints.md`：本轮约束的决策记录。

## 硬性规则

- 仓库内项目文档默认使用中文编写；引用外部原文、协议字段、代码标识符和错误输出可以保留原文。
- 使用 PowerShell 读取项目文档时，`Get-Content` 必须显式加 `-Encoding UTF8`，避免中文内容在控制台中乱码。
- 不在框架未定阶段引入具体技术栈、包管理器、构建工具或运行时，除非已有经过记录的 ADR。
- 所有业务输入、外部 API、配置、环境变量、存储读写结果和跨端通信都必须在边界处进行类型化解析或显式校验。
- 后台、Web、App 与共享能力之间的依赖方向必须清楚、单向、可检查。
- 重要设计、取舍、缺口和临时债务必须沉淀到 `docs/`，不能只留在聊天记录或个人记忆里。
- 新增代码必须带有相称的验证方式。没有可运行测试时，必须说明替代验证步骤和剩余风险。
- 不复制粘贴临时工具函数到多个端；稳定共性能力应进入共享层或明确记录不共享的原因。

## 工作方式

- 小改动可以直接实现；跨端、跨模块或会影响架构边界的改动，需要先新增或更新计划/ADR。
- 需要计划的工作，必须把计划落实到 `docs/plans/` 下的本地文件，不能只保留在聊天记录中。
- Symphony、Linear 等外部任务系统可以作为主计划来源；如果外部任务已有完整步骤，本地计划不机械复制产品/项目管理步骤，但必须记录 repo 相关实现状态、验证缺口、技术债、风险、决策和 commit 关联。
- 当工作跨模块、跨多次提交或跨多天，涉及架构边界、ADR、技术选型、安全或高风险行为，存在验证缺口、技术债、回滚条件或复杂失败处理，外部任务不足以让后续智能体从仓库恢复状态，外部任务系统不可访问，或用户明确要求时，必须在 `docs/plans/active/` 创建或更新本地计划。
- 发现文档与代码不一致时，把不一致当作缺陷处理：要么修正文档，要么修正代码。
- 当约束阻碍实现时，不绕过约束；先更新设计记录，说明为什么需要调整。
- 输出应优先让后续智能体可读：明确入口、边界、验证命令和失败处理。
