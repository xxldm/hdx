# 文档分层边界

本文是仓库文档放置规则的事实源。当前文档体系按维护范围分成两套项目文档：

- 根仓库 `docs/`：公开项目文档，覆盖除后端内部实现以外的内容。
- `services/backend/docs/`：后端项目文档，写法与根仓库项目文档一致，但范围只限后端内部内容。

本仓库不再维护额外的 internal docs 层。能公开的内容放根仓库 `docs/`；后端内部内容放 `services/backend/docs/`。

## 根仓库 `docs/`

根仓库可以放：

- 入口、约束、文档边界和智能体工作规则。
- 跨端架构、公开 ADR、公开计划、技术债和工程操作说明。
- Web、Desktop、App、共享层、公开契约和端侧交互边界。
- 部署者需要知道的配置模板、环境变量、外部工具准备项、服务名、Nacos Data ID、OpenAPI 快照名和 release asset 名。
- 已讨论但尚未进入计划、ADR、接口草案、代码或测试的非后端内部结论，临时放 `docs/discussions/`。

根仓库不要放：

- 后端内部调用链、模块间调用关系、适配器位置、公共工具模块职责拆分和接入方式。
- 后端数据库表字段、索引、migration 路径、Repository/JPA/JDBC 细节。
- 后端基础设施接入方式、native/AOT 诊断、验证流水账、内部契约和实现计划细节。
- 密钥、令牌、证书、真实用户数据或真实生产配置。

## 后端 `services/backend/docs/`

后端文档按同一套项目文档规则维护，但只覆盖后端范围。可以放：

- 后端内部 ADR、计划、技术债和实现讨论。
- 后端模块边界、调用关系、内部契约、适配器设计和基础设施接入方式。
- 数据库迁移、实体、仓储、SQL、字段、索引、软删除、审计和后端验证。
- Native Image、AOT、依赖裁剪、诊断报告和后端 release resolver 细节。
- 已讨论但尚未进入后端计划、ADR、接口草案、代码或测试的后端结论，临时放 `services/backend/docs/discussions/`。

后端文档同样不要放密钥、令牌、证书、真实用户数据或真实生产配置。

## 计划和讨论

`docs/plans/` 保留 repo 级计划、公开实现状态、验证摘要、剩余风险和入口路由。计划触及后端内部细节时，公开计划只写摘要和后端文档入口，具体内容写入 `services/backend/docs/plans/`。

`docs/discussions/` 和 `services/backend/docs/discussions/` 只是临时缓冲区：

- 只放已经讨论确认、但还没有正式承接位置的结论。
- 按主题拆小文件，不写大而全的产品说明书。
- 一旦内容进入计划、ADR、接口草案、代码、测试或对应实现文档，就从讨论目录删除或改成很短的入口。
- 不记录完整聊天流水、反复过程和一次性命令输出；需要长期解释取舍时，写 ADR。

## 配置边界

部署者需要知道的配置放根仓库 `docs/`，例如 Nacos 模板、外部工具准备项、环境变量示例、公开服务名、端口和 release 包配置方式。

只有后端实现者需要知道的配置放 `services/backend/docs/`，例如配置如何被后端模块读取、内部适配器如何接入、native/AOT 特殊处理、内部开关迁移策略和诊断参数。

## 判断问题

写文档前先问：

1. 用户、部署者或非后端端侧维护者是否需要知道？如果是，优先放根仓库 `docs/`。
2. 这条内容是否只对后端实现者有意义，或能帮助复刻后端内部结构？如果是，放 `services/backend/docs/`。
3. 这条内容是否已经由计划、ADR、接口草案、代码、测试或 README 承载？如果是，不要重复写一份。
4. 这条内容是否只是还没落正式位置的讨论结论？如果是，临时放对应范围的 `discussions/`。
5. 这条内容是否只是过程日志，而不是当前事实？如果是，收敛或归档，不继续堆到 active plan。

## 维护要求

- 新增长期规则时，优先进入约束、架构、ADR、计划或对应端 README，不新增长期产品说明书。
- 新增后端内部实现草案时，放 `services/backend/docs/`，公开主仓库只保留入口链接。
- 如果发现同一条规则在多个位置重复，必须指定事实源；其他位置只保留摘要或链接。
- 再次裁剪公开 git 历史前，必须先确认当前公开文档边界和后端承接位置。
