# ADR 0001：采用智能体友好的工程约束

- 日期：2026-05-25
- 状态：已接受

## 背景

HDX 当前是一个框架未定的工具箱项目，暂定包含后台服务端、Web 端和 App 端。由于后续开发可能大量依赖编码智能体，需要在选型和实现前先建立清晰边界、事实源和验证回路。

OpenAI 的 Harness Engineering 实践强调：人类负责明确意图、设计环境和反馈回路，智能体在可读、可验证、边界清晰的仓库中执行。该思路适合本项目当前阶段。

## 决策

采用以下约束作为项目初始工程底座：

- 仓库内项目文档默认使用中文。
- `AGENTS.md` 保持简短，只做地图和硬规则。
- `docs/` 作为项目事实源，记录约束、架构、质量门禁、计划和 ADR。
- 框架和技术栈未定前，不引入具体实现依赖。
- 后台、Web、App 和共享层的依赖方向先行固定。
- 跨边界数据必须解析或校验。
- 重要取舍、临时债务和验证缺口必须版本化。

## 影响

- 后续任何技术选型都需要 ADR。
- 后续代码生成必须服从 `docs/CONSTRAINTS.md` 和 `docs/ARCHITECTURE.md`。
- 文档维护成为开发完成标准的一部分。
- 未来可逐步把这些约束转成 lint、测试和 CI。

## 备选方案

- 先选技术栈再补约束：速度更快，但容易在早期形成难以检查的隐性边界。
- 只写 README：入口简单，但不足以承载跨端项目和智能体协作所需的上下文。
- 写一个很长的智能体说明文件：短期集中，但容易过期，也会挤占智能体上下文。

## 验证方式

短期通过人工检查文档结构与新增 ADR 执行情况。框架确定后，将逐步加入自动化检查。
