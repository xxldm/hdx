# ADR 0005：服务端会话撤销使用 Redis

- 日期：2026-06-06
- 状态：已接受
- 修订：2026-07-05 收敛为公开安全策略；具体声明字段、Redis key、适配器、TTL 计算、读写入口和验证细节以后端私有文档为准。

## 背景

HDX 服务端认证中心使用 JWT/OAuth2。JWT access token 是自包含令牌，如果受保护接口只校验签名和过期时间，用户登出后旧 access token 在过期前仍然可以访问受保护接口。

用户确认 HDX 需要登出、强制下线和高风险会话失效尽快生效，并采用 “JWT + 服务端撤销状态” 策略。Redis 作为服务端共享低延迟存储参与该策略；本 ADR 只记录公开安全决策，不记录具体字段、key 结构、适配器位置或验证流水账。

## 决策

- access token 继续使用 JWT。
- 服务端必须维护可共享的会话撤销状态，使登出、强制下线、refresh token 重放和高风险会话失效能在 access token 自然过期前生效。
- Redis 作为第一阶段服务端会话撤销状态的共享存储；Redis 地址等非密钥配置可以通过公开 Nacos 模板表达，Redis 密码通过环境变量或部署平台 Secret 注入。
- 受保护请求必须经过受控后端入口并遵守撤销状态；已撤销或无效的会话不得继续访问受保护接口。
- Redis 或等效撤销存储不可用时，受保护请求必须 fail-closed，返回服务不可用语义，不得静默放行。
- 单 token 粒度撤销暂不实现；后续有更细撤销粒度需求时再补充设计。
- 具体 JWT 声明字段、Redis key、TTL 计算、适配器位置、读写入口、错误码映射和测试类清单以后端私有文档为准。

## 备选方案

- 只缩短 access token 有效期并撤销 refresh token：实现简单，但登出不是即时生效。
- 按单个 token 拉黑：粒度更细，但一次登录会话可能经过 refresh 产生多个 access token，登出场景需要撤销整个会话，当前阶段复杂度更高。
- PostgreSQL 撤销表：功能可行，但 gateway 每次请求查关系数据库不适合作为高频认证路径。
- opaque token + introspection：撤销语义更集中，但会让每次资源访问依赖认证中心或 introspection 缓存，当前阶段复杂度更高。

## 影响范围

- 后端认证入口和受保护业务入口必须共同遵守会话撤销策略。
- Redis 属于服务端部署需要准备的外部工具，公开部署文档可以说明需要 Redis、Nacos 模板和密钥注入方式。
- Redis 配置细项、Nacos Data ID、适配器位置、key 结构、TTL 计算和服务端验证命令以后端私有文档为准；公开主仓库只记录密钥不得提交、密码走环境变量或部署 Secret 的边界。

## 验证方式

- 后端私有测试覆盖未撤销会话放行、已撤销会话拒绝、无效令牌拒绝和撤销存储不可用时 fail-closed。
- 后端私有集成验证覆盖登录、登出、旧令牌失效和服务端撤销存储异常路径。
- 公开主仓库只验证对外错误语义、OpenAPI/前端契约和部署边界，不记录具体测试类或命令流水账。

## 回滚条件

- 业务确认不再要求登出即时生效，只接受短 access token 的时间窗口。
- 目标部署环境无法提供 Redis 或等效低延迟共享存储。

回滚时需移除服务端会话撤销检查、撤销写入和 Redis 相关配置，并把策略退回为短 access token + refresh token 撤销。

## 后续事项

- 完成真实服务端 profile 迁移验证，确认认证相关表创建正常；表名和迁移细节以后端私有文档为准。
- Web BFF 登出时调用认证中心登出，并清理自身 session/cookie。
