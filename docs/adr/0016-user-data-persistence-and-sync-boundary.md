# ADR 0016：用户数据持久化与跨端同步边界

- 日期：2026-06-23
- 状态：已接受

## 背景

Web 工作台布局已经从浏览器 `localStorage` 主状态转向后端持久化。随着主题、语言、菜单固定、组件配置、计时器预设和后续真实模块继续增加，用户数据如果继续由各端临时决定存储位置，会让同步、冲突、离线和 Desktop Full 边界变得难以维护。

HDX 需要一份长期章程，先规定不同数据的事实源、离线能力和跨端边界，再让各模块在此基础上扩展自己的存储模型。

## 决策

### 数据分层

用户相关数据按语义分层，不能都塞进 layout、config 或单一杂项表。

- 账号级偏好：语言、主题、菜单固定、工作台布局、组件显示偏好等，默认需要跨 Web、Desktop Online 和移动端 App 同步。
- 工具模块业务数据：真实工具模块自己的内容、设置和记录，由模块拥有独立契约和存储边界。
- 组件布局数据：只保存位置、尺寸、外观 chrome、header 显示项、内容 key 和必要配置引用，不保存模块业务内容。
- 设备级状态：窗口大小、托盘、自启动、远端地址、运行中的计时器状态、临时 UI 展开状态等，只属于当前设备。
- 安全敏感运行态：access token、refresh token、本机 sidecar token 等只留在各自安全运行边界，不进入普通用户数据同步。

计时器需要拆开处理：

- 计时器预设、默认时长列表和组件内展示配置属于用户数据，可以进入后端同步。
- 计时器运行状态属于设备级状态，包括是否运行、暂停、剩余时间、结束时间和响铃状态，不做跨设备同步。

### 后端自建表基础字段

除 Spring Authorization Server 等第三方框架官方 schema 外，HDX 自建、用户可感知、会被修改、删除或同步的可变业务表，默认使用表内版本、审计和软删除字段：

```text
id
version
created_at
created_by_user_id
updated_at
updated_by_user_id
deleted
deleted_at
deleted_by_user_id
```

规则：

- `*_by_user_id` 存稳定用户 ID，不存用户名、昵称或显示名。
- 默认不增加 `actor_type`。远端服务端用户使用认证用户 ID；Desktop Full 可使用保留本机用户 ID，例如 `local-admin`；系统任务可使用保留用户 ID，例如 `system`。
- 用户名、昵称或显示名只在查询投影或冲突响应中生成，不能作为审计事实字段。
- 软删除字段优先放在原业务表内，不新建删除记录表。删除操作只设置 `deleted=true`、写入删除时间和删除人，并递增 `version`。
- 程序不做物理删除和自动清理。需要清理时，由管理员按单独 runbook 或数据库管理流程手动处理。
- 普通业务查询默认过滤 `deleted=false`。唯一约束需要按“未删除记录唯一”设计，避免软删除数据阻塞同名重建。
- 追加型日志、审计、事件、outbox 和纯技术表可以明确例外。此类表通常不修改、不删除，操作者或来源应作为业务字段表达，例如 `actor_user_id`、`operation`、`source` 或 `producer`。
- 关联表不默认豁免。只要关联关系本身是业务事实，并且需要被授予、撤销、同步或审计，就按可变业务表处理，例如 `user_role`、`role_permission`、团队成员、工具归属关系。
- 纯从属父对象、可由父对象整体重建、没有单独审计价值的明细表可以简化，例如工作台布局快照中的 widget 明细行。此类表的冲突和审计由父对象整体版本负责。
- 现有历史表如果使用 `created_by`、`updated_by`、`deleted_by` 等旧命名，后续修改相关 schema 时必须评估迁移到本 ADR 命名，或记录明确例外。

### Web 与 Desktop Online

Web Online 和 Desktop Online 的登录用户数据以远端后端为事实源。

- 已登录但后端接口不可用时，不能静默回退默认布局或旧本地数据；界面应显示不可用空态和错误提示。
- Web 和 Desktop Online 不进入弱网/无网离线写边界；页面缓存按普通在线页面处理。
- 浏览器和 WebView 不保存敏感 token，不把后端地址、数据库配置或服务端密钥暴露给前端运行时。
- `localStorage` 不作为登录用户数据主存储；短期只允许用于非敏感、可丢弃、设备级 UI 暂存，并且不能影响后端事实源。

### 移动端 App 离线与跨设备同步

移动端 App 仍保持 Online first，但第二阶段允许弱网、无网下暂存白名单内的可同步用户修改。

- 移动端 App 可以保存本地缓存、白名单离线操作和待同步队列，联网后通过后端同步提交。
- 移动端 App 不规划完整离线业务引擎，不内置本机 HTTP 后端服务。
- 移动端 App 离线能力采用白名单；只有客户端能本地完成校验、保存和展示结果的功能才允许离线操作。
- 依赖后端生成、权限判断、公开治理、通知正式投递或多人协作的功能，离线时不可编辑，只能只读展示缓存或提示联网后使用。
- 移动端 App 离线修改必须记录同步所需元数据，例如本地操作 ID、目标对象、基础版本、设备 ID、创建时间和幂等键。
- 移动端 App 重连后由后端负责校验版本、幂等和权限；冲突不能只在客户端本地悄悄覆盖。

跨设备同步是产品目标。Web、Desktop Online 和移动端 App 应围绕同一远端账号数据模型工作；移动端 App 允许暂存并晚提交，不代表它拥有另一套独立事实源。

### Desktop Full

Desktop Full 包含本机数据模式，使用本机后端和本机数据库作为本机数据事实源。

- Desktop Full 的业务数据、用户数据、工作台布局、组件配置和模块数据进入本机数据库，而不是 Tauri app config。
- Tauri app config 只管理纯客户端配置，例如开机自启、远端地址、窗口偏好、托盘偏好和本机能力开关。
- Desktop Full 的本机数据以本机后端和本机数据库为事实源；Desktop Full 也可以连接已登录服务端和匿名公开来源，但各来源数据仍归各来源所有，不自动同步或合并。本机数据与服务端账号之间的受控在线迁移见 ADR 0019。
- Desktop Full 相当于本机有一个常驻可用服务，不按弱网/无网暂存能力处理。
- Desktop Online 与 Desktop Full 不在同一运行时随意切换；两个 flavor 的数据边界保持清楚。

### 冲突处理原则

冲突策略尚需按真实模块细化，但先接受以下底线：

- 需要跨设备同步的可变记录必须有服务端版本或等价修订号，客户端提交时携带基础版本。
- 后端发现基础版本过期时，不能默认最后写入获胜；应返回明确冲突响应，由客户端进入冲突处理流程。
- 默认冲突粒度是模块内的业务数据记录，不是模块整体。模块负责定义数据类型、字段语义、API 和展示摘要；记录负责持有 `id`、`version` 和软删除状态。
- 只有天然整体对象才使用整体版本，例如工作台布局、排序快照、整份全局配置。此类对象内部局部变化会影响整体语义。
- 低价值偏好可以按字段或对象做简单覆盖，但删除、内容编辑、排序、布局和模块业务数据不能无提示覆盖。
- 列表新增、审计追加、计时器预设新增等天然可合并操作，应优先设计成幂等命令或追加事件，减少人工冲突。
- 模块可以声明自己的合并策略，但必须通过后端契约表达，不能只藏在某个端的 UI 代码里。

后续优先讨论三类冲突：

- 账号级偏好：语言、主题、菜单固定等可以倾向最后一次有效保存，但应保留版本以避免旧客户端覆盖新 schema。
- 布局与排序：倾向使用整体版本校验；冲突时让用户选择保留服务器版本、覆盖服务器版本或另存为草稿。
- 工具模块业务数据：默认在记录级使用乐观锁和人工处理；没有明确合并策略时，不使用静默覆盖。

冲突响应由公共协议和模块摘要共同组成：

- 公共协议返回 `code`、`resourceType`、`resourceId`、`baseVersion`、`currentVersion`、更新时间和更新人 ID。
- 模块提供 `resourceLabel`、`serverSummary` 或类似摘要，帮助 UI 生成“谁把哪条数据改成了什么”的提示。
- 覆盖不能使用无条件 `force=true`。用户确认覆盖时，客户端必须基于冲突响应中的 `currentVersion` 再提交；如果期间又被修改，则继续返回冲突。

## 备选方案

- 所有端都直接使用本地存储，后续再同步：前期快，但会继续放大 SSR hydration、幽灵布局、跨设备不一致和迁移成本。
- 移动端 App 也禁止离线修改：实现简单，但与手机端弱网、无网暂存诉求冲突。
- Desktop Full 使用 Tauri app config 保存全部数据：实现轻，但 config 更适合少量客户端设置，不适合承载大量业务数据、索引、迁移和查询。
- 全部冲突使用最后写入获胜：实现最简单，但会静默丢数据，尤其不适合布局、排序、删除和模块内容。

## 影响范围

- 后端需要按用户和模块继续扩展账号级偏好、工作台、组件配置和模块数据 API。
- Web Online 继续通过 Nuxt BFF 访问后端，不恢复 `localStorage` 作为用户数据主状态。
- Desktop Online 通过 Rust BFF 访问远端后端；Desktop Full 通过本机 sidecar 和本机数据库保存用户数据。
- 移动端 App 后续工程需要为本地缓存、白名单离线操作、同步队列、冲突提示和加密存储单独设计。
- `packages/shared` 后续应沉淀跨端可共享的数据契约、错误码、同步状态码和冲突响应类型。

## 验证方式

本 ADR 只固定边界，不直接实现同步队列或本机数据库迁移。移动端离线暂存、写操作幂等、本地提醒兜底和冲突 UI 的未归档讨论结论见 `docs/discussions/offline-queue-conflict-and-reminders.md`。文档变更按 `docs/QUALITY.md` 执行。

后续实现时至少需要补充：

- 后端：版本校验、幂等提交、冲突响应和权限隔离测试。
- Web：接口不可用空态、保存失败、冲突提示和不读取用户数据 `localStorage` 的回归测试。
- Desktop Full：本机数据库迁移、在线迁移校验和 app config 只存客户端配置的测试或手工检查。
- 移动端 App：白名单离线操作落库、重放同步、幂等、冲突保留和加密存储验证。

## 回滚条件

满足以下任一情况时，需要新增 ADR 替代本决策：

- 移动端 App 放弃弱网、无网暂存修改，改为严格在线。
- Desktop Full 不再使用本机后端和本机数据库，或需要自动与远端账号双向同步。
- 后续出现统一同步框架或 CRDT 方案，能显著降低冲突处理复杂度并满足当前多端边界。

## 后续事项

- 设计同步响应错误码和冲突响应结构，进入 shared/OpenAPI 契约。
- 设计移动端 App 离线同步队列、加密存储、重试和冲突 UI。
- 按 ADR 0019 实现本机数据与服务端账号之间的在线迁移、迁移后状态和来源连接状态。
- 梳理现有主题、语言、菜单固定、计时器预设和组件配置，逐步从临时端侧状态迁到对应事实源。
- 每个新工具模块接入前，先声明它的数据是否账号级同步、模块级同步、设备级状态或纯临时状态。
