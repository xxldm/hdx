# ADR 0018：Desktop Full/Online 备份导入导出边界

- 日期：2026-07-04
- 状态：已接受

## 背景

Desktop Full 使用本机后端和本机数据库，Desktop Online 连接服务端并把数据保存在服务端。两者是不同安装包形态，不在同一个运行时内切换。

用户可能从 Full 换到 Online，或从 Online 换到 Full。为了卸载、换客户端形态或更换设备时不丢数据，需要手动导入导出能力。但本能力不应演变成同步、迁移或跨后端合并。

## 决策

- Desktop Full 和 Desktop Online 都需要支持导出和导入备份包。
- Full 与 Online 导出的备份包可以互相导入：
  - Full 导入到本机后端和本机数据库。
  - Online 导入到当前登录账号所在的当前服务端。
- 导入导出是用户主动触发的数据备份/搬家能力，不是自动同步、迁移、合并或跨后端复制。
- 不做跨数据源 ID 保留。导入时目标端生成新 ID，原记录 ID 只可作为排障来源信息保存。
- 第一版只做追加导入，不做覆盖导入，不删除或覆盖目标端已有数据。
- 导入目标永远是当前客户端当前数据源；Online 导入时必须先连接服务端并完成登录。

## 备份包格式

备份包扩展名为 `.hdxbak`，本质是 zip 容器。第一版结构：

```text
manifest.json
data/
  todos.ndjson
  schedules.ndjson
  rules.ndjson
  ...
checksums.json
attachments/
  ...
```

规则：

- `manifest.json` 记录 `formatVersion`、导出应用版本、导出时间、来源模式、数据范围、领域 schema version、记录数量和是否包含附件。
- `data/*.ndjson` 按领域拆分，每行一条记录，支持流式读写，避免大备份一次性载入内存。
- `checksums.json` 记录各文件 SHA-256 和大小，导入前必须校验。
- `attachments/` 预留给后续文件能力；第一版可以没有附件。
- 导出永远使用当前最新格式，不提供导出旧格式。

## 版本兼容

- 当前应用必须能导入同版本和旧版本备份。
- 旧版本备份通过显式转换器升级到当前导入模型。
- 遇到未来 `formatVersion` 或不支持的领域 schema version 时，拒绝导入并提示用户升级客户端。
- 包格式版本与领域 schema version 分开维护，避免某个领域变化导致全包格式无谓升级。

## 导出范围

- Full 可导出本机后端内当前支持的用户业务数据。
- Online 只能导出当前登录账号有权限导出的数据。
- 组织/空间数据是否可导出由后续空间权限设计决定；第一版不自动导出当前账号无权管理或无权导出的空间数据。
- 不导出 token、会话、密码、系统角色、权限授予记录、发布权限、公开头衔或治理/审核记录。
- 不导出公开数据，包括公开主页配置、公开事项/公开日程发布状态、公开流、关注、订阅、公开权限、维护者关系、下架、申诉和公开参与者展示关系。
- 如果某条私有日程或 todo 曾由公开事项关联而来，导出时只按用户自己的私有副本处理；不导出原公开来源、关注关系或发布状态。

## 导入规则

- 导入前读取 `manifest.json`，校验版本、数据范围、文件 checksum 和记录数量。
- 导入前展示摘要，例如 todo、日程、规则、空间等记录数量，以及可能被跳过的内容。
- 导入时按领域分批处理，记录导入批次 ID，失败时能给出已导入、跳过和失败分类。
- 导入公开来源的私有副本时，导入结果仍是普通私有 todo 或日程。
- Online 导入只能写入当前账号有权限的个人范围或目标空间；空间目标选择和权限规则后续按空间设计细化。
- 通知设置可以随业务记录导入，但通知计划默认不直接启用；是否重新启用由用户确认或后续模块规则决定。

## 备选方案

- 一个巨大 JSON 文件：实现简单，但数据多、附件多或版本变动后难以流式读写和断点定位，不采用。
- Full/Online 自动同步：体验顺滑，但会引入冲突、权限、空间归属、公开数据和多后端一致性问题，不采用。
- 导入时保留原数据库 ID：看似方便排查，但不同后端 ID 空间不一致，容易破坏目标端引用和唯一约束，不采用。
- 默认覆盖导入：容易误删或覆盖已有数据，第一版不采用。

## 影响范围

- Desktop capability 层需要提供导出文件、选择备份文件、进度展示和错误展示能力。
- Full 导出/导入通过本机 sidecar 完成。
- Online 导出/导入通过当前服务端账号接口完成。
- 后端需要为可导出领域提供稳定导出模型和导入命令，不能直接暴露数据库表结构。
- `packages/shared` 后续可承载 `.hdxbak` manifest schema、领域导入导出 schema 和错误码。

## 验证方式

- 文档验证：`pwsh -NoLogo -NoProfile -File scripts/quality-gate.ps1 -Scope docs -NoBuild`。
- 后续实现时至少需要：
  - 备份包 manifest/checksum 校验测试。
  - 大文件 NDJSON 流式导入导出测试。
  - Full 导出后 Online 导入、Online 导出后 Full 导入的互通测试。
  - 旧版本备份升级测试。
  - 未来版本拒绝导入测试。
  - 公开数据、token、会话、权限授予记录不进入备份包的检查。

## 回滚条件

- 用户确认 Full 与 Online 不再需要互相导入导出。
- 项目改为统一账号同步或云备份，并通过新的 ADR 接受对应冲突、权限和隐私边界。
- `.hdxbak` zip + NDJSON 无法满足目标平台文件能力或性能要求，需要替换为数据库快照或其他容器格式。
