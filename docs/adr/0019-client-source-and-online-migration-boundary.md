# ADR 0019：客户端多来源与在线迁移边界

- 日期：2026-07-08
- 状态：已接受
- 替代：ADR 0018

## 背景

Desktop Full 后续不只是单纯“本机离线客户端”。本机模式也可能连接多个服务端和公开来源；Web、Desktop Online、App 也会面对多个服务端或公开订阅来源。

如果继续使用 `.hdxbak` 备份包在 Full 和 Online 之间导入导出，会让“迁移、备份、跨来源复制、同步”边界混在一起。项目需要一条更清晰的规则：多来源可以聚合查看，但不做跨来源同步；确实需要离开本机或离开服务端时，只提供受控在线迁移。

## 决策

### 多来源模型

- 客户端可以连接多个平级来源，不设主账号/次账号。
- 来源类型包括本机数据、已登录服务端和匿名只读公开来源。
- 主导航按 Todo、日程、通知、规则等业务模块组织，不按后端或空间拆分。
- 来源和空间只作为轻量标识、筛选条件和状态信息出现。
- 右上角提供来源状态入口，展示各来源在线、离线、登录失效、已迁出和上次同步等状态。
- 某个来源不可用时，不打断整个页面；其他来源继续可用。当前筛选只看不可用来源时，显示该来源的离线或不可用空态。

### 保存位置

- 客户端有一个全局 `默认保存位置`，普通新增、首页快捷新增和从公开事项生成个人 Todo / 日程默认写入该位置。
- 用户在选择来源时可以勾选 `设为默认保存位置`；未勾选时只对本次操作生效。
- 默认保存位置不可写、离线或权限失效时，操作时重新弹出来源选择。
- 空间上下文和规则生成内容不使用全局默认保存位置；它们写入所属来源。
- 只读公开来源不能作为保存目标。

### 不支持跨来源

- 不支持跨来源移动、同步、智能合并或普通内容复制。
- 不把某个来源的数据搬到另一个来源作为日常功能。
- 用户把公开事项加入个人 Todo / 日程时，是在默认保存位置创建个人安排，不视为跨来源复制原数据。
- 规则生成的 Todo / 日程永远跟随规则所在来源。

### 在线迁移

只支持两种在线迁移方向：

- 本机数据 -> 已登录服务端。
- 已登录服务端 -> 本机数据。

不支持：

- 后端对后端迁移。
- 离线迁移包。
- `.hdxbak`。
- 普通备份导出。
- 服务端数据反复下载。

### 本机 -> 服务端

- 本机客户端登录目标服务端后，直接上传本机业务数据。
- 目标服务端只按目标账号/空间/实例配额判断是否允许迁入；不暴露单次大小上限或分批迁移给用户。
- 迁移频率可以配置，例如每 X 分钟/小时/天/月最多 N 次。
- 迁入成功后清理本机业务数据，避免聚合视图里出现本机和远端两份重复内容。
- 清理范围包括 Todo、日程、规则、候选池、通知偏好和本地声音等本机业务数据。
- 不清理连接配置、登录信息、公开订阅源和客户端布局设置。
- 迁移成功后必须生成一条站内通知进入收件箱，不受用户通知偏好影响；该通知不发声音、桌面通知或强制横幅。
- 迁移完成可以显示 Toast；Toast 是操作反馈，不算通知渠道。

确认文案：

```text
迁移成功后，本机中的这些内容会从列表中移除，之后请在目标服务端继续使用。以后如需从该服务端迁回本机，每个账号只能操作一次，完成后服务端账号将被禁用。确定迁移吗？
```

### 服务端 -> 本机

- 本机客户端登录服务端后，直接拉取该账号可迁出的个人数据到本机。
- 一个服务端账号只允许迁出到本机一次。
- 迁出成功后，本机导入为本机数据，服务端账号进入不可恢复的已迁出状态。
- 服务端账号不能继续登录、编辑、同步、迁入、再次迁出或恢复。
- 客户端中的该后端连接状态变为 `已迁出`，不再参与聚合，不再作为保存目标，不允许换账号登录或重新认证恢复。
- 用户只能移除此连接记录；如需使用同一服务端的其他账号，先移除连接再重新添加。
- 本机导入成功后，必须在本机收件箱生成一条站内通知留痕，不受用户通知偏好影响；服务端账号已不可继续登录，不再向该账号发普通通知。
- 确认页还需说明：迁出只删除已迁出的个人数据，未删除的数据里该用户显示为 `已注销用户`，系统会保留邮箱等基础账号信息用于禁止重复注册和安全审计。

连接详情文案：

```text
该服务端账号已迁出到本机，服务端账号不可继续使用。你可以移除此连接记录。
```

确认文案：

```text
迁出完成后，这些内容会导入本机数据，服务端账号将被禁用，之后只能在本机继续使用。确定迁出吗？
```

### 迁移不携带的内容

两种方向都不迁移：

- 登录态、token、会话。
- 权限授权记录。
- 公开关系、公开主页和公开发布治理关系。
- 协作关系。
- 通知投递历史。

服务端迁出或账号注销只删除个人数据。未被删除的数据中，原用户统一显示为 `已注销用户`。

## 备选方案

- 继续使用 `.hdxbak` 备份包：能离线搬家，但会让备份、迁移和跨来源复制混在一起，且容易被误用为日常同步，不采用。
- 支持后端对后端拉取：体验更顺，但两个服务端不一定互信，会扩大信任边界，不采用。
- 支持跨来源复制/移动：看似灵活，但会引入权限、公开状态、协作关系和通知计划归属问题，不采用。
- 本机 -> 服务端 上传后保留本机业务数据：会导致聚合视图重复并让用户继续改错来源，不采用。

## 影响范围

- Desktop Full 需要同时支持本机数据和连接多个服务端/公开来源。
- Web/Desktop/App 的主导航和列表聚合不按来源拆分，但需要来源筛选和来源状态入口。
- 账号生命周期需要支持 `migrated_out` 或等价不可恢复状态。
- 认证、注册和配额策略需要防止通过无限注册绕过迁移和存储限制。
- 旧的 `.hdxbak` manifest、NDJSON 导入导出和普通备份导出方案不再作为当前实现目标。

## 验证方式

本文只固定边界，不直接实现迁移。后续实现时至少需要覆盖：

- 多来源筛选、不可用来源空态和来源状态入口。
- 默认保存位置选择、失效和只读来源不可选。
- 本机 -> 服务端 成功后本机业务数据清理、强制站内通知留痕和失败不清理。
- 服务端 -> 本机 每账号一次、客户端连接进入已迁出状态、不能换号登录。
- 迁移不携带公开、协作、权限授权、登录态和通知投递历史。

## 回滚条件

- 项目决定重新引入普通备份或离线迁移包。
- 项目决定支持跨来源移动、复制、同步或合并。
- Desktop Full 不再允许连接多个服务端或公开来源。
