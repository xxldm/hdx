# 执行计划

本目录用于存放较复杂工作的 repo-local 计划、进度、验证缺口和技术债记录。它的目标不是替代项目管理系统，而是让后续智能体能从仓库本身恢复实现状态。

## 目录

- `active/`：进行中的计划。
- `completed/`：已完成的计划。
- `tech-debt-tracker.md`：已知技术债。

## 与外部任务系统的关系

- Symphony、Linear 等外部任务系统可以作为主计划来源，用于展示完整计划步骤、负责人、状态、排期和产品/项目管理信息。
- 如果外部任务已有完整步骤，本地计划不需要机械复制全部产品或项目管理步骤。
- 仓库内计划文件负责记录与 repo 直接相关的交接信息：实现状态、验证缺口、技术债、风险、决策和 commit 关联。
- 外部任务系统不能完全替代 repo-local handoff；后续智能体必须能在无法访问外部系统时，从仓库判断当前实现进度、验证结果和剩余风险。
- 本地计划可以引用 Symphony/Linear 链接或编号，并只摘录对代码、文档、验证和回滚有直接影响的内容。

## 何时必须创建本地计划

符合以下任一条件时，必须在 `docs/plans/active/` 创建计划文件：

- 工作跨模块、跨多次提交或跨多天。
- 涉及架构边界、ADR、技术选型、安全或高风险行为。
- 存在验证缺口、技术债、回滚条件或复杂失败处理。
- 外部任务内容不足以让后续智能体从仓库恢复实现状态。
- 外部任务系统对后续智能体不可访问。
- 用户明确要求本地计划。

## 何时可以不创建本地计划

同时满足以下条件时，可以不创建本地计划：

- 工作范围小、单一、可在一次提交内完成。
- 不涉及架构边界、ADR、技术选型、安全或高风险行为。
- 没有需要跟踪的验证缺口、技术债、复杂回滚或失败处理。
- 外部任务不存在，或外部任务已经足以承担项目管理信息且不影响 repo-local 交接。
- 用户没有明确要求本地计划。

未创建本地计划时，最终回复仍需说明变更范围、验证结果和剩余风险。

## 本地计划最低内容

本地计划至少包含：

- Symphony/Linear 等外部任务链接或编号；没有外部任务时写明“无”。
- 当前状态。
- repo 内实现范围。
- 本地代码、文档和验证 checklist。
- 状态记录。
- 验证结果。
- 剩余风险。
- 相关 commit。

推荐使用 `docs/plans/TEMPLATE.md` 创建新计划。

## 状态显示与同步

- 计划状态必须使用 Markdown checkbox，或使用包含“状态”列的表格。
- 主要步骤完成、遇到阻塞、验证完成、最终回复前，都必须同步更新计划状态和状态记录。
- 验证缺口、临时债务、风险、回滚条件和相关 commit 发生变化时，也必须更新本地计划。
- 完成后的计划应移动到 `docs/plans/completed/`，并保留关键决策和验证结果。
- 如果完成后仍需保留在 `active/`，必须在计划中明确记录原因和下一次清理条件。
